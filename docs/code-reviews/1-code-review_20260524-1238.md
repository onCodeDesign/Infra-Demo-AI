## Remarks

### ЁЯЯб Warnings

#### R1: `CustomerService` missing explicit `internal` modifier

- **File:** `Modules/Sales/Sales.Services/CustomerService.cs`
- **Commit:** `533f025`
- **Line(s):** L12
- **Dimension:** Architecture Compliance
- **Description:** The class is declared as `class CustomerService` without an access modifier, making it implicitly `internal`. The project convention (`.github/copilot-instructions.md`) requires `internal` to be stated explicitly: *"Mark services `internal` unless explicitly exported via `Contracts`."* Implicit visibility is easy to overlook when the file is later refactored.
- **Suggested Fix:** Change `class CustomerService` тЖТ `internal class CustomerService`.

### ЁЯЯв Suggestions

#### R2: XML doc comments on `OverdueCustomerSummary` properties restate the obvious

- **File:** `Modules/Contracts/Sales/OverdueCustomerSummary.cs`
- **Commit:** `b6a6542`
- **Line(s):** L5-L7, L10-L12, L14-L16
- **Dimension:** Code Quality
- **Description:** The project convention states *"Only comment code that needs a bit of clarification. Do not comment otherwise."* The three `<summary>` blocks (`"Full name of the customer: "{FirstName} {LastName}""`, `"Total number of overdue orders for this customer"`, `"Due date of the oldest overdue order for this customer"`) add no information beyond the property names themselves. The names `CustomerName`, `OverdueOrderCount`, and `OldestOverdueDueDate` are self-documenting.
- **Suggested Fix:** Remove the XML doc comments and rely on the property names to convey intent. If the `{FirstName} {LastName}` format truly needs documenting, a single-line comment on `CustomerName` is sufficient.

---

#### R3: Ordering test asserts names only, not the date values driving the sort

- **File:** `Modules/Sales/Sales.Services.UnitTests/CustomerServiceTests.cs`
- **Commit:** `83fde6f`
- **Line(s):** L87-L111
- **Dimension:** Test Quality
- **Description:** `GetCustomersWithOverdueOrders_OrderedByOldestOverdueDueDate_Ascending` verifies the result sequence via `CustomerName` only. The test is valid because the data is arranged so that alphabetical name order coincides with ascending date order, but the causal link between `OldestOverdueDueDate` and the sort position is not asserted. A future refactor that accidentally sorts by name would leave this test green.
- **Suggested Fix:** In addition to `CustomerName`, include `OldestOverdueDueDate` in the expected objects and use `options.Including(x => x.CustomerName).Including(x => x.OldestOverdueDueDate).WithStrictOrdering()`, so the test pins both the sort key and the output field.

### тД╣я╕П Notes

#### R4: High-level design Mermaid shows `GetEntities<Customer>()` but implementation uses `GetEntities<SalesOrderHeader>()`

- **File:** `Modules/Sales/Sales.Services/CustomerService.cs`
- **Description:** The sequence diagram in `docs/workitems/1-design.md` shows `Svc->>Repo: GetEntities<Customer>()`, but the implementation queries from `SalesOrderHeader` and groups up to the customer. Both approaches are semantically correct; the detailed design text explicitly specifies *"Filter: SalesOrderHeaders where DueDate < today AND Status is open"*, which matches the implementation. The diagram is a minor inaccuracy in the HLD document, not a code defect. No code change needed.

---

#### R5: `FakeRepository` is hand-rolled rather than an NSubstitute substitute

- **File:** `Modules/Sales/Sales.Services.UnitTests/CustomerServiceTests.cs`
- **Description:** The test uses a private `FakeRepository : IRepository` backed by a `List<object>` instead of `Substitute.For<IRepository>()`. This is the correct choice тАФ `CustomerService.GetCustomersWithOverdueOrders()` executes LINQ-to-Objects against the result of `GetEntities<T>()`, and NSubstitute cannot transparently forward LINQ expression trees to in-memory data. The hand-rolled fake is appropriate and follows the spirit of the unit-testing conventions (stub named `FakeRepository`).
